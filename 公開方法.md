公開は大きく **GitHub Pages** か **Netlify** の2択です。どちらも「GitHubにpush → 自動ビルド → 公開」になります。

---

## 方法A: GitHub Pages（GitHubだけで完結・おすすめ）

### 1) GitHub側の設定

リポジトリ → **Settings → Pages → Source を “GitHub Actions”** にします。([GitHub Docs][1])

### 2) Workflowを追加（Hugo公式の例でOK）

リポジトリにこのファイルを作成：
`.github/workflows/hugo.yaml`

中身（最小構成例）：

```yaml
name: Deploy Hugo site to Pages

on:
  push:
    branches: ["main"]
  workflow_dispatch:

permissions:
  contents: read
  pages: write
  id-token: write

concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v5
        with:
          submodules: recursive
          fetch-depth: 0

      - name: Setup Pages
        id: pages
        uses: actions/configure-pages@v5

      - name: Install Dart Sass
        run: sudo snap install dart-sass

      - name: Install Hugo
        run: |
          curl -sLJO "https://github.com/gohugoio/hugo/releases/download/v0.152.2/hugo_extended_0.152.2_linux-amd64.tar.gz"
          sudo tar -C /usr/local/bin -xf hugo_extended_0.152.2_linux-amd64.tar.gz hugo
          rm hugo_extended_0.152.2_linux-amd64.tar.gz

      - name: Build
        run: |
          hugo --gc --minify --baseURL "${{ steps.pages.outputs.base_url }}/"

      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./public

  deploy:
    runs-on: ubuntu-latest
    needs: build
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

* Anatoleはテーマを **submodule** で入れているので、`actions/checkout` の `submodules: recursive` が重要です。([gohugo.io][2])
* SCSSを使うので **Dart Sass** が必要（あなたが出たエラーのGitHub版対策）です。([gohugo.io][2])
* デプロイは `actions/deploy-pages` が「artifactをPagesに出す」公式アクションです。([GitHub][3])

### 3) pushして確認

`main` にpushすると Actions が走り、成功すると Pages のURLが表示されます（Hugo公式もその流れで案内）。([gohugo.io][2])

## 方法B: Netlify（UIが分かりやすい・独自ドメインも楽）

### 1) `netlify.toml` を追加（Hugo公式例）

リポジトリのルートに `netlify.toml` を作ります。Hugo公式が例を載せています：([gohugo.io][4])

```toml
[build.environment]
  HUGO_VERSION = "0.152.2"
  DART_SASS_VERSION = "1.93.2"

[build]
  publish = "public"
  command = """\
    curl -sLJO "https://github.com/sass/dart-sass/releases/download/${DART_SASS_VERSION}/dart-sass-${DART_SASS_VERSION}-linux-x64.tar.gz" && \
    tar -C "${HOME}/.local" -xf "dart-sass-${DART_SASS_VERSION}-linux-x64.tar.gz" && \
    rm "dart-sass-${DART_SASS_VERSION}-linux-x64.tar.gz" && \
    export PATH="${HOME}/.local/dart-sass:${PATH}" && \
    git config core.quotepath false && \
    hugo --gc --minify --baseURL "${URL}"
    """
```

### 2) NetlifyでGitHub連携してImport → Deploy

Netlifyダッシュボードから「既存プロジェクトのImport」を選んで進める手順が、Hugo公式にステップで載っています。([gohugo.io][4])

## どっちを選ぶべき？

* **まずは無料・シンプルに公開したい** → GitHub Pages
* **独自ドメイン/フォーム/リダイレクト等も触りそう** → Netlify

今のリポジトリ名が「`<username>.github.io`」かどうかで、公開URL（User Page / Project Page）が変わります。もし希望があれば、その前提で **baseURLや多言語（/en/）込みの最終チェック項目**だけ短く整理します。


はい、**GitHub Pagesでも独自ドメイン（例：`example.com` / `www.example.com`）は使えます**。([GitHub Docs][1])

## 手順（GitHub Pages + Hugo + GitHub Actions想定）

### 1) 先に「どの形で公開するか」を決める

* **Apex（ルート）**：`example.com`
* **サブドメイン**：`www.example.com`（おすすめにされることが多い）

GitHub側は、apex なら `A/ALIAS/ANAME`、サブドメインなら `CNAME` で設定します。([GitHub Docs][1])

---

### 2) GitHub側でカスタムドメインを設定

リポジトリ → **Settings → Pages → Custom domain** に、使いたいドメインを入力して **Save**。([GitHub Docs][1])

※あなたは GitHub Actions で公開しているはずなので、この場合 **`CNAME` ファイルは作られず、既存の `CNAME` ファイルも無視されます（不要）**。([GitHub Docs][1])

---

### 3) ドメイン管理会社（DNS）でレコードを設定

#### 3-A) apex（`example.com`）で使う場合

DNSで `A` レコード（`@`）を GitHub Pages のIPに向けます：([GitHub Docs][1])

* `185.199.108.153`
* `185.199.109.153`
* `185.199.110.153`
* `185.199.111.153`

（任意）IPv6を使うなら `AAAA` も設定できます：([GitHub Docs][1])

* `2606:50c0:8000::153`
* `2606:50c0:8001::153`
* `2606:50c0:8002::153`
* `2606:50c0:8003::153`

#### 3-B) `www.example.com` などサブドメインで使う場合

DNSで `CNAME` を作って、`www` を **`<user>.github.io`（リポジトリ名は付けない）**に向けます。([GitHub Docs][1])

例：

* `www.example.com` → `<your-username>.github.io`

---

### 4) apex と www を両方使いたい（おすすめ構成）

GitHubは **apex を使う場合、`www` もセットアップすることを推奨**していて、両方のDNSが正しく設定されていれば **自動リダイレクト**も行われます。([GitHub Docs][1])

---

### 5) HTTPSを有効化

リポジトリ → **Settings → Pages → Enforce HTTPS** をON。
カスタムドメインでもHTTPSとHTTPS強制に対応し、DNSチェック後に証明書（Let’s Encrypt）が自動発行されます。([GitHub Docs][2])

---

## セキュリティ：ドメイン検証（強く推奨）

独自ドメインは、状態によっては「ドメイン乗っ取り」リスクがあるので、GitHubの **ドメイン検証（TXTレコード）**をしておくのが安全です。([GitHub Docs][3])
手順は「GitHubのプロフィール設定 → Pages → Add a domain → 指示されたTXTレコードをDNSに追加 → Verify」です。([GitHub Docs][3])

---

## 注意点（詰まりやすいところ）

* `*.example.com` みたいな **ワイルドカードDNSは非推奨**（乗っ取りリスク）。([GitHub Docs][1])
* `www` を apex に「転送」する設定（プロバイダのWeb転送）ではなく、**Pagesの推奨どおりCNAME/Aで**組むのが安全です（HTTPSで問題が出やすい）。([GitHub Docs][1])
* 公開URLが確定したら、Hugoの `baseURL` も `https://example.com/`（または `https://www.example.com/`）に合わせると、リンク生成が安定します。

---

もし、あなたが使いたい独自ドメインが **`example.com`（apex）** なのか **`www.example.com`** なのかだけ教えてくれたら、DNSレコードを「そのままコピペで設定できる形（ホスト名/種別/値）」にして書き起こします。
